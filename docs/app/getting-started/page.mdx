# Getting Started

import { Steps, Callout, FileTree, Tabs } from 'nextra/components'

Dockflow deploys to a single server or a multi-node cluster — same workflow. All you need to get started is one Linux server.

<Steps>
## Install the CLI

<Tabs items={['npm (any OS)', 'npx (no install)', 'Linux / macOS', 'Windows']}>
  <Tabs.Tab>
    ```bash
    npm install -g @dockflow-tools/cli
    dockflow --version
    ```
  </Tabs.Tab>
  <Tabs.Tab>
    Run any command directly without installing:

    ```bash
    npx @dockflow-tools/cli <command>
    ```

    For example:

    ```bash
    npx @dockflow-tools/cli deploy production
    ```

    <Callout type="info">
      Pin a specific version with `npx @dockflow-tools/cli@1.2.3 <command>`.
    </Callout>
  </Tabs.Tab>
  <Tabs.Tab>
    ```bash
    curl -fsSL https://raw.githubusercontent.com/Shawiizz/dockflow/main/install.sh | bash
    dockflow --version
    ```
  </Tabs.Tab>
  <Tabs.Tab>
    ```powershell
    irm https://raw.githubusercontent.com/Shawiizz/dockflow/main/install.ps1 | iex
    dockflow --version
    ```
  </Tabs.Tab>
</Tabs>

## Setup Your Server

You need a Debian/Ubuntu server with SSH access (root or sudo user).

<Tabs items={['Interactive (Recommended)', 'Non-Interactive']}>
  <Tabs.Tab>
    If the CLI is installed **on the server**:

    ```bash
    dockflow setup
    ```

    If the CLI is installed **on your local machine**:

    ```bash
    dockflow setup root@your-server-ip
    ```

    The wizard handles everything: Docker installation, Swarm initialization, SSH keys, and deployment user creation.
    At the end, it generates a **connection string** — save it for later.
  </Tabs.Tab>
  <Tabs.Tab>
    For automated provisioning (Terraform, CI, etc.). Must be run **on the target server**:

    ```bash
    dockflow setup \
      --host $(hostname -I | awk '{print $1}') \
      --user dockflow \
      --password "your-secure-password" \
      --generate-key \
      --portainer \
      --portainer-password "portainer-admin-password"
    ```

    See `dockflow setup --help` for all options.
  </Tabs.Tab>
</Tabs>

## Create Your Project

```bash
dockflow init
```

This generates the following structure at the root of your project:

<FileTree>
  <FileTree.Folder name=".dockflow" defaultOpen>
    <FileTree.File name="config.yml" />
    <FileTree.File name="servers.yml" />
    <FileTree.Folder name="docker" defaultOpen>
      <FileTree.File name="docker-compose.yml" />
      <FileTree.File name="accessories.yml" />
      <FileTree.File name="Dockerfile" />
    </FileTree.Folder>
    <FileTree.Folder name="hooks" />
  </FileTree.Folder>
</FileTree>

- **`config.yml`** — Project name, build strategy, health checks ([reference](/configuration))
- **`servers.yml`** — Servers and environment variables ([reference](/configuration/servers))
- **`docker/`** — Docker Compose stack, accessories, and Dockerfile
- **`hooks/`** — Pre/post build and deploy hook scripts ([reference](/configuration#hooks))

## Deploy

<Tabs items={['CLI', 'CI/CD']}>
  <Tabs.Tab>
    Create a `.env.dockflow` file at the root of your project with the connection string(s) from setup:

    ```bash
    PRODUCTION_MAIN_SERVER_CONNECTION=eyJ...base64...
    ```

    Then deploy:

    ```bash
    dockflow deploy production
    ```

    <Callout type="info">
      The `.env.dockflow` file is loaded automatically by the CLI. Do not commit it to your repository.
    </Callout>
  </Tabs.Tab>
  <Tabs.Tab>
    ### Workflow file

    The CLI auto-detects the environment and version from your CI provider's variables (tag name, branch, commit SHA). No external workflow dependency needed.

    You can generate these files automatically with `dockflow init`.

    <Tabs items={['GitHub Actions', 'GitLab CI', 'Bitbucket Pipelines']}>
      <Tabs.Tab>
        Create `.github/workflows/dockflow-build.yml` and `.github/workflows/dockflow-deploy.yml`:

        ```yaml copy
        # .github/workflows/dockflow-build.yml
        name: Build

        on:
          push:
            branches: ['*']

        jobs:
          build:
            runs-on: ubuntu-latest
            steps:
              - uses: actions/checkout@v4
              - run: npx @dockflow-tools/cli build
                env:
                  DOCKFLOW_SECRETS: ${{ toJSON(secrets) }}
        ```

        ```yaml copy
        # .github/workflows/dockflow-deploy.yml
        name: Deploy

        on:
          push:
            tags: ['*']

        jobs:
          deploy:
            runs-on: ubuntu-latest
            steps:
              - uses: actions/checkout@v4
              - run: npx @dockflow-tools/cli deploy
                env:
                  DOCKFLOW_SECRETS: ${{ toJSON(secrets) }}
        ```

        <Callout type="info">
          Pin a specific CLI version with `npx @dockflow-tools/cli@1.2.3`.
        </Callout>
      </Tabs.Tab>

      <Tabs.Tab>
        Create `.gitlab-ci.yml` at the root of your project:

        ```yaml copy
        stages:
          - build
          - deploy

        # CI: Build images on branch push (needs Docker)
        build:
          stage: build
          image: node:24
          services:
            - docker:dind
          variables:
            DOCKER_HOST: tcp://docker:2375
            DOCKER_TLS_CERTDIR: ""
          before_script:
            - apt-get update -qq && apt-get install -y -qq docker.io
          script:
            - npx @dockflow-tools/cli build
          rules:
            - if: $CI_COMMIT_TAG
              when: never
            - when: always

        # CD: Deploy on tag push (SSH-only, no Docker needed)
        deploy:
          stage: deploy
          image: node:24
          script:
            - npx @dockflow-tools/cli deploy
          rules:
            - if: $CI_COMMIT_TAG
        ```
      </Tabs.Tab>

      <Tabs.Tab>
        Create `bitbucket-pipelines.yml` at the root of your project:

        ```yaml copy
        image: node:24

        definitions:
          services:
            docker:
              image: docker:dind

        pipelines:
          # CI: Build images on branch push (needs Docker)
          branches:
            '*':
              - step:
                  name: Build
                  services:
                    - docker
                  script:
                    - npx @dockflow-tools/cli build

          # CD: Deploy on tag push (SSH-only, no Docker needed)
          tags:
            '*':
              - step:
                  name: Deploy
                  script:
                    - npx @dockflow-tools/cli deploy
        ```

        <Callout type="info">
          Bitbucket Pipelines reads secrets from **Repository variables** — go to Repository Settings → Repository variables to add them.
        </Callout>
      </Tabs.Tab>
    </Tabs>

    ### Repository secrets

    Add the connection string generated during setup to your CI/CD secrets:

    <Tabs items={['Connection String (Recommended)', 'Individual Secrets']}>
      <Tabs.Tab>
        | Secret Name | Description | Example |
        |-------------|-------------|---------|
        | `[ENV]_[SERVERNAME]_CONNECTION` | Connection string per server | `PRODUCTION_MAIN_SERVER_CONNECTION` |
        | `GIT_TOKEN` | Optional: for remote build strategy | GitHub/GitLab token |

        Where to add secrets:
        - **GitHub**: Settings → Secrets and variables → Actions
        - **GitLab**: Settings → CI/CD → Variables
        - **Bitbucket**: Repository Settings → Repository variables

        <Callout type="error">
          **Do not share connection strings** — they contain SSH private keys.
        </Callout>
      </Tabs.Tab>

      <Tabs.Tab>
        | Secret Name | Description | Example |
        |-------------|-------------|---------|
        | `[ENV]_[SERVERNAME]_SSH_PRIVATE_KEY` | SSH private key | `PRODUCTION_MAIN_SERVER_SSH_PRIVATE_KEY` |
        | `[ENV]_[SERVERNAME]_HOST` | Server IP/hostname | `PRODUCTION_MAIN_SERVER_HOST` |
        | `[ENV]_[SERVERNAME]_USER` | SSH user (default: dockflow) | `PRODUCTION_MAIN_SERVER_USER` |
        | `[ENV]_[SERVERNAME]_PORT` | SSH port (default: 22) | `PRODUCTION_MAIN_SERVER_PORT` |
        | `[ENV]_[SERVERNAME]_PASSWORD` | Sudo password (optional) | `PRODUCTION_MAIN_SERVER_PASSWORD` |
        | `GIT_TOKEN` | Optional: for remote build strategy | GitHub/GitLab token |
      </Tabs.Tab>
    </Tabs>

    <Callout type="warning">
      Server names use underscores in secrets. A server named `main_server` becomes `MAIN_SERVER` in secret names.
    </Callout>
  </Tabs.Tab>
</Tabs>
</Steps>
